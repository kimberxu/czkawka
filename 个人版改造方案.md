# 个人版 Krokiet 线程控制改造方案 (Windows/HDD 优化)

本文档描述了针对 Windows 个人使用环境（混合 SSD/HDD，多物理硬盘）的定制化改造方案。
核心目标是解决 Czkawka 原生“全局线程池”在机械硬盘上导致的磁头争用（Thrashing）和非均衡负载问题。

## 一、 核心架构：基于物理磁盘的信号量 (Physical Disk Semaphores)

放弃依赖原生的全局线程数设置，引入一套基于文件路径硬编码的**旁路 IO 控制系统**。

### 1. 硬件拓扑识别 (硬编码策略)
在代码中建立盘符到物理磁盘 ID 的映射（需根据实际机器配置修改）：

| 盘符路径 | 物理设备类型 | 内部 Disk ID | 并发配额 (Limit) |
| :--- | :--- | :---: | :---: |
| `C:\`, `D:\` | SSD (NVMe) | **0** | **999** (无限) |
| `E:\` | HDD (机械盘 A) | **1** | **1** (严格串行) |
| `F:\` | HDD (机械盘 B) | **2** | **1** (严格串行) |

### 2. 控制逻辑
引入全局静态信号量集合。在涉及磁盘 IO 的热点代码路径中，必须先申请对应 Disk ID 的令牌（Permit）才能进行读取操作。

## 二、 各扫描模式改造细节

### 1. 重复文件扫描 (Duplicate Finder)
*   **目标**：**绝对的单线程顺序 IO**。
*   **痛点**：原版在 Pre-hash 阶段（读取文件头 2KB）如果是多线程，会导致磁头剧烈跳跃。
*   **改造方案**：
    *   **Pre-hash**: 在读取前 2KB 数据的代码块外围，申请 IO 锁。
    *   **Full-hash**: 在读取全量文件的代码块外围，申请 IO 锁。
*   **行为**：
    *   即使 Rayon 开启 16 线程，针对 E 盘永远只有一个线程在读数据。
    *   读取完成后立即释放锁，计算哈希（CPU）不占用锁。

### 2. 相似图片扫描 (Similar Images)
*   **目标**：**IO 串行，CPU 并行 (流水线化)**。
*   **痛点**：原版 `image::open` 混合了 IO 和解码，锁住整个过程会浪费 CPU。
*   **改造方案**：
    *   **重构加载流程**：将“读取”与“解码”拆分。
    *   **Step 1 (IO)**: 申请 IO 锁 -> `fs::read(path)` -> 释放 IO 锁。
    *   **Step 2 (Compute)**: `image::load_from_memory(buffer)` (无锁，全速并行)。
*   **行为**：
    *   HDD 磁头始终在顺序读取下一张图片（1 线程）。
    *   CPU 利用空闲线程并行解码已读入内存的图片。

### 3. 相似视频扫描 (Similar Videos)
*   **目标**：**受控的粗粒度并发**。
*   **痛点**：视频处理涉及多次 Seek 和 Decode 交替，无法拆分。
*   **改造方案**：
    *   **外层拦截**：在处理单个视频的最外层循环申请锁。
    *   **特殊配额**：针对 HDD，将信号量 Limit 设为 **2**。
*   **行为**：
    *   允许最多 2 个视频同时处理，以平衡 IO 等待（Seek）和 CPU 负载。
    *   防止 16 个视频同时被打开导致硬盘假死。

## 三、 解决的核心问题

1.  **非均衡负载陷阱 (Imbalanced Load)**
    *   *问题*：E 盘 100G，F 盘 10G。原版在 F 盘扫完后，所有线程会围攻 E 盘，导致 E 盘性能雪崩。
    *   *解决*：改造后，F 盘扫完释放线程，但 E 盘的信号量限制为 1，多余线程会被阻塞（挂起），E 盘始终保持单线程最高效运行。

2.  **SSD/HDD 混用**
    *   *问题*：为了迁就 HDD 开 1 线程，导致 SSD 性能跑不满。
    *   *解决*：Disk 0 (SSD) 的配额为无限。C 盘文件扫描可以跑满 CPU，而 E 盘文件扫描依然乖乖排队。

## 四、 开发指引 (Action List)

1.  **新建模块**：`disk_control.rs`，实现 `get_disk_id` 和 `DiskSemaphores`。
2.  **Duplicate Core**: 修改 `hash_calculation`，注入 `with_io_lock`。
3.  **Image Core**: 重写 `image::open` 为 `read` + `load`，在 `read` 处注入锁。
4.  **Video Core**: 在处理循环头部注入锁 (Limit=2)。
