# Czkawka Agent Guidelines

This document provides essential instructions for AI agents working on the Czkawka codebase.
Follow these guidelines to ensure code consistency, stability, and performance.

## 1. Project Overview

Czkawka is a multi-crate Rust workspace for finding duplicate files and other lint:
- `czkawka_core`: Core logic and algorithms (Focus for logic changes).
- `czkawka_cli`: Command-line interface.
- `czkawka_gui`: GTK4-based GUI.
- `krokiet`: Slint-based GUI.

**Key Config Files:**
- `justfile`: Primary task runner.
- `clippy.toml`: Linting configuration.
- `czkawka_core/Cargo.toml`: Core dependencies.

## 2. Build, Test, and Lint Commands

Always prioritize `just` commands if available.

### Standard Commands
- **Build All**: `just build_all` (Runs release build, clippy, tests).
- **Run CLI**: `just run <args>` (e.g., `just run -h`).
- **Fix & Format**: `just fix` (Runs `cargo fmt`, `clippy --fix`, and Python `ruff`).
- **Clean**: `cargo clean`.

### Running Tests
- **Run All Tests**: `cargo test` (or `just test` implied in `build_all`).
- **Run Single Test**:
  ```bash
  cargo test -p czkawka_core --lib <test_name> -- --nocapture
  ```
  *Example*: `cargo test -p czkawka_core --lib test_hash_calculation`

### Linting
- **Check**: `cargo clippy --all-targets --all-features`.
- **Configuration**: Strictly follow `clippy.toml`.
  - `unwrap_used = "warn"`: Avoid `unwrap()`. Use `?` or `expect()` with context.
  - `indexing_slicing = "warn"`: Be careful with `[]`. Use `get()` or verify bounds.
  - `print_stdout = "warn"`: Use `log::info!` or `println!` only in CLI implementation.

## 3. Code Style & Conventions

### General Rust Patterns
- **Formatting**: Must pass `cargo fmt`.
- **Naming**: `snake_case` for functions/vars, `CamelCase` for types.
- **Imports**: Group imports logically. Use `crate::` for internal references.
- **Visibility**: Prefer `pub(crate)` over `pub` where possible to minimize public API surface.

### Error Handling
- Use **`Result<T, String>`** or **`anyhow::Result`** (check file context).
- Avoid `panic!`. Propagate errors.
- **Logging**: Use `log` crate (`debug!`, `info!`, `error!`).
  - Example: `log::debug!("Processing file: {:?}", path);`

### Architecture
- **Separation of Concerns**:
  - `czkawka_core` should NOT depend on GUI crates.
  - CLI/GUI crates handles user interaction and calls `czkawka_core`.
- **Tools**: New features usually go into `czkawka_core/src/tools/`.

## 4. Concurrency & Threading Limits (CRITICAL)

Czkawka relies heavily on **Rayon** for performance.

### Guidelines
1.  **Initialization**:
    - The global Rayon thread pool is initialized in `czkawka_core/src/common/mod.rs` via `set_number_of_threads`.
    - It sets the worker thread stack size to **4 MB** (`DEFAULT_WORKER_THREAD_SIZE`).
    - Defaults to system parallelism if not specified.

2.  **Limiting Threads**:
    - **User/Runtime**: Respect `RAYON_NUM_THREADS` environment variable.
    - **In Tests**: If writing a test that spawns threads, consider limiting them:
      ```rust
      // Example of setting thread limit for specific scope (if needed)
      let pool = rayon::ThreadPoolBuilder::new().num_threads(2).build().unwrap();
      pool.install(|| {
          // code running with limited threads
      });
      ```

3.  **Responsiveness & Granularity**:
    - Use `.with_max_len(n)` (e.g., 2 or 3) on parallel iterators to ensure frequent checks for stop signals.
    - This is critical for the GUI to remain responsive when cancelling long operations.

4.  **Memory Usage**:
    - Note `THREAD_BUFFER_SIZE` (default 2MB) in `duplicate/mod.rs` for data buffering.
    - Massive thread counts can spike memory usage (N threads * (4MB stack + 2MB buffer)).

5.  **Safety**:
    - Use `AtomicBool` for stop flags (`stop_flag: &Arc<AtomicBool>`).
    - Use `AtomicU64` for shared counters (size, progress).
    - Avoid `Mutex` in hot loops; prefer message passing (`crossbeam_channel`) or atomics.

## 5. Development Workflow

1.  **Verify First**: Run `just fix` before committing.
2.  **Add Tests**: Unit tests go in the same file (usually `mod tests`) or `tests/`.
3.  **Documentation**: Update `Instruction.md` if changing logic visible to users.
4.  **No Regressions**: Ensure `cargo test` passes 100%.

## 6. Common Issues & Solutions

- **"Too many open files"**: When scanning deep directories, ensure `file_handler` is dropped or use `ulimit` in Linux.
- **CI Failures**: Often due to `clippy` warnings. Treat warnings as errors.
- **Path Handling**: Use `PathBuf` and `to_string_lossy()` carefully. Windows/Unix differences matter.
